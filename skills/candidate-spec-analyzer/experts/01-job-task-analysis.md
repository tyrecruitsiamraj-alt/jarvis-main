# Expert 1 — Job/Task Analysis

## บทบาท
แยกแยะข้อมูลดิบในใบขอให้เป็นโครงสร้างที่ใช้งานได้ และอ่านบริบทแฝงที่ฟอร์มไม่ได้เขียนตรง ๆ

## วิธีวิเคราะห์

### ขั้นที่ 1 — แยกฟิลด์เป็น 4 กลุ่ม

| กลุ่ม | ตัวอย่างฟิลด์ | บทบาท |
|---|---|---|
| Identity/Metadata | เลขที่ใบขอ, ผู้ส่ง, วันที่ส่ง, ผู้ติดต่อหน่วยงาน | Track งาน ไม่เกี่ยวกับสเปคคน |
| Hard Filter | วันที่ต้องการ, จำนวน, สถานที่, ประเภทใบขอ, รายได้, ลักษณะงาน | เงื่อนไขตัดสิทธิ์ |
| Soft Scoring | ช่วงอายุ, เพศ (ถ้าระบุ), รายละเอียดตำแหน่ง | จัดอันดับหลังผ่าน Hard Filter |
| Context | ทำงานวันสุดท้าย, ชื่อ/สาเหตุคนลาออก, วันเวลาเข้างาน | บริบทประกอบการตัดสินใจ |

### ขั้นที่ 2 — อ่านบริบทแฝง (สิ่งที่ต้องตรวจทุกครั้ง)

1. **ประเภทใบขอ = "เพิ่มอัตรา" vs "ทดแทนคนลาออก":**
   - ทดแทน → ดึงสเปคคนเดิม/สาเหตุลาออกมาวิเคราะห์เพิ่ม (เช่น ลาออกเพราะเงินเดือนน้อย → ระวังปัญหาเดิมซ้ำ)
   - เพิ่มอัตรา → ไม่มีคนเดิมอ้างอิง ต้องตีสเปคจากบริบทงานล้วน ๆ

2. **ช่องว่างระหว่างวันที่ต้องการกับวันที่ยื่นใบขอ:**
   - วันที่ต้องการมาก่อนวันที่ยื่น → ขอย้อนหลัง/เร่งด่วนผิดปกติ → ต้องระบุใน Operations expert (2.6)

3. **"ตำแหน่ง (รายละเอียด)" ที่เป็นคำเฉพาะขององค์กร** เช่น "แรงงานสำรอง", "Rotation Staff", "Standby" — มักหมายถึงคนที่ไม่ผูกไซต์ตายตัว ต้องยืนยันสัดส่วน/เงื่อนไขกับหน่วยงาน

4. **"เพศ: ไม่ระบุ"** → ต้อง handle เป็นค่า ANY ไม่ใช่ค่าที่ต้อง exact match; **ถ้าระบุเพศเฉพาะ** ให้ส่งต่อประเด็นนี้ให้ Labor Compliance Expert (2.4) ตรวจสอบ BFOQ

5. **"รายได้"** ในใบขอ = อัตราที่หน่วยงานเสนอ ไม่ใช่เงินเดือนที่ผู้สมัครคาดหวัง → ต้อง match แบบมี tolerance ไม่ใช่ exact (ส่งต่อให้ Compensation Expert 2.3)

6. **"วันเวลาเข้างาน" แบบ free text** (เช่น "ตามตารางกะ กะละ 9 ชม.") → normalize เป็น shift_type + hours_per_shift แยกกัน ส่งต่อให้ Operations Expert (2.6)

### ขั้นที่ 3 — ระบุ Job Family เบื้องต้น (ส่งต่อให้ STEP 1 ของ SKILL.md ยืนยันอีกครั้งกับ taxonomy)
